# microriv-soc Architecture & Design Log (Phase 3)

This document captures the architectural decisions, memory mappings, and clock domain crossing (CDC) design log for Phase 3 of the **microriv-soc** project.

---

## 1. System Block Diagram & Dual-Clock Topology

Phase 3 splits the SoC into two distinct clock domains running at different frequencies:
1. **CPU Clock Domain (`clk_cpu`)**: Runs at 50 MHz (20ns period), clocking the PicoRV32 CPU core, system SRAM, and the APB Master Bridge.
2. **Peripheral Clock Domain (`clk_periph`)**: Runs at 12.5 MHz (80ns period), clocking the APB decoding logic, UART, GPIO, and Timer peripherals. The peripheral clock is generated by a divide-by-4 synchronous clock divider (`clk_divider.v`) inside the top-level module.

```

|                                                                        |
|  +--------------+                      +-------------------------+     |

|  |     Core     | -------------------> |      (APB3 Master)      |     |
|  +-------+------+                      +------------+------------+     |
|          | (SRAM Bus)                               |                  |
|          v                                          v                  |
|     +----+----+                             +-------+-------+          |
|     | SRAM RAM|                             |  cdc_bridge   |          |

+---------------------------------------------+-------+-------+----------+
```

## 3. CDC Bridge Design: Level Handshake vs Asynchronous FIFO

### Rationale for Choosing Level Handshake:
1. **CPU Bus Blocking Model**: The PicoRV32's native memory bus blocks instruction execution until `mem_ready` is asserted. It is a single-outstanding-transaction bus. The CPU cannot issue multiple pipelined or back-to-back writes in a burst. Thus, the write-buffering advantage of an Asynchronous FIFO is completely unutilized.
2. **Resource and Area Efficiency**: A level handshake requires only two 2-stage synchronizers (a total of 4 flip-flops) to cross control handshake signals, plus holding registers for address and data. An asynchronous FIFO requires dual-port memory cells, gray-code address counters, and synchronizers for read and write pointers, which takes significantly more silicon area and increases verification complexity.
3. **Data Path Safety (Glitches and Skew)**: Under a level handshake control protocol, the address and data buses are held completely static and stable in the source domain before the request (`req`) is synchronized. They remain stable throughout the entire transaction until the acknowledge (`ack`) is received. This guarantees that multi-bit bus skew will not result in corrupted transactions.

## 4. APB CDC Bridge Timing and Latency Analysis

Every peripheral access crossing the CDC boundary incurs the following round-trip latency overhead:

1. **Request Synchronization**:
   - `req_cpu` is asserted in the CPU domain.
   - It takes **2 cycles** of the destination `clk_periph` domain to synchronize `req_cpu` -> `req_periph_sync` using a 2-stage synchronizer.
2. **APB Transaction Execution**:
   - **1 cycle** of `clk_periph` for the Setup Phase.
   - **1 cycle** of `clk_periph` for the Access Phase (assuming a 0-wait-state slave like GPIO or Timer).
3. **Acknowledge Synchronization**:
   - Once `PREADY_periph` is high, `ack_periph` is asserted.
   - It takes **2 cycles** of `clk_cpu` to synchronize `ack_periph` -> `ack_cpu_sync` back to the CPU domain.
4. **Clean-up Phase**:
   - Once the CPU domain sees `ack_cpu_sync` high, it clears `req_cpu`.
   - It takes **2 cycles** of `clk_periph` to synchronize `req_cpu` = 0 back to the peripheral domain.
   - The peripheral domain then clears `ack_periph`.

**Total round-trip latency for a single 0-wait-state peripheral access**:
* CPU Domain: 2 cycles (Setup + Access) + 2 cycles synchronization = **4 clk_cpu cycles**.
* Peripheral Domain: 2 cycles synchronization + 2 cycles APB transaction = **4 clk_periph cycles**.
* Because `clk_periph = clk_cpu / 4`, 4 `clk_periph` cycles equals 16 `clk_cpu` cycles.
* Therefore, a complete APB peripheral write/read transaction takes approximately **20 clk_cpu cycles**. This is the latency cost of crossing clock boundaries safely using handshakes.

## 5. CDC Verification & Interview Diagnostic Log

### "What would happen if the CDC synchronization were removed?"
If the control signals (`PSEL`, `PENABLE`, `PREADY`) and data/address buses crossed between the 50 MHz domain and the 12.5 MHz domain directly without synchronizers:

1. **Metastability Risk on Control Pins**:
   If the clock edges of `clk_cpu` and `clk_periph` occur close to each other, a control input (like `PSEL`) transitioning in the CPU domain would violate setup/hold times at the peripheral receiver flip-flops. The receiver flip-flop output would hover in an unstable, intermediate voltage state (metastability) for an indeterminate time. If this metastable value resolves to different logical states at different destination registers, the peripheral state machines would enter invalid or illegal states, locking up the bus.
2. **Bus Skew Corruption**:
   A 32-bit bus (such as `PADDR` or `PWDATA`) contains 32 separate wires. Due to variations in physical trace routing delays (skew), different bits arrive at slightly different times. If the peripheral domain samples the bus without control handshaking while the bits are still changing, it will read transient, corrupted values. The handshake protocol solves this by ensuring all data lines are static and stable long before they are read.
